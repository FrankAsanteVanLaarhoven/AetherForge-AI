# v2.34 — Tool-Call Format Control

Milestone type: **inference-time control experiment** (NO training). **Decision: HOLD/REJECT** per the
strict gate — with a decisive diagnostic (below).

Branch: `feature/v234-tool-call-format-control`.

## Scientific purpose

v2.34 isolates tool-call emission/control because **v2.31, v2.32, and v2.33 all failed primarily
through `no_tool_call` collapse** on the frozen 32-task benchmark (v2.33: adapter 5/32, `no_tool_call`
26/32 dominant). This is **not repair training**; it is an inference-time, deterministic
scaffold-control experiment answering: *can we recover valid execute_code/tool-call emission before
any further adaptation?*

The real failure mechanism (from the v2.33 transcripts): the model writes `TOOL_CALL:` followed by raw
code/asserts but never the `execute_code({...})` schema the agent parser needs — a genuine tool-use
intent scored as `no_tool_call`.

## Components

- `scripts/tool_call_format_control.py` — pure, deterministic controller (no model calls): detect a
  valid tool call, detect `no_tool_call`, detect invalid tool JSON, extract the model's intended code,
  and repair/wrap a near-miss into the canonical `execute_code({"code": ...})` schema **only when
  unambiguous** (iterative `TOOL_CALL`→`CHANGED_TOOL_CALL` = latest-wins; conflicting different-source
  candidates = rejected). It **never invents code** — it only re-frames code the model already emitted,
  so it cannot fabricate an unsafe tool call.
- `scripts/eval_v234_tool_call_format_control.py` — offline counterfactual over the v2.33 32-task
  transcripts (baseline = scaffold adapter without control; controlled = same outputs with the
  controller). Recovered passes are verified against the **benchmark's own assertions** (parsed from
  each task's `Verify:` clause); the model's self-serving `print('PASS')` is stripped and never
  trusted.
- `scripts/summarise_v234_tool_call_format_control.py::decide` — the promotion gate.

## Result (offline counterfactual, deterministic, reproducible)

| Metric | baseline (v2.33, no control) | controlled (v2.34) |
|---|---|---|
| 32-task pass | 5/32 | **5/32** |
| tool_call_rate | 0.188 | **0.969** |
| no_tool_call | 26 | **1** |

- Recovered tool-calls: **25**; recovered passes: **0**; unsafe/ambiguous repairs rejected: **1**.
- `no_tool_call` dominant after control: **False** (shifts to `wrapped_no_passing_solution` = 24).
- tree_serialize preserved: **True** (1/1).

## Decision: HOLD/REJECT

Gates: `no_tool_call_not_dominant` PASS, `tool_call_rate_improves` PASS, `tree_serialize_preserved`
PASS, `no_unsafe_fabrication` PASS, `artifact_safety` PASS — but **`score_improves_over_5of32` FAIL**
(5→5). Per the strict gate, **HOLD/REJECT**.

**Diagnostic (the value of v2.34):** the tool-call FORMAT/emission bottleneck is **resolvable at
inference time** (no_tool_call 26→1, tool-call rate 0.19→0.97, 25 calls recovered), but the 32-task
score is unchanged because the recovered calls wrap asserts/incomplete code without a correct solution.
**The bottleneck has shifted from tool-call emission to solution generation.** This redirects the next
milestone away from scaffold/emission and toward producing a correct solution body.

## Claim boundary

v2.34 makes **no** claim of model improvement, repair improvement, SOTA, SWE-bench success, or
production readiness. It is a deterministic control/measurement experiment; no new generation was
performed (it reuses existing local benchmark transcripts); no model, champion adapter, or memory
index was created or overwritten. The controller only re-wraps code the model already emitted, and
recovered passes are verified against the benchmark's real tests — so no result is fabricated.
