# v2.17 Dense Retrieval Pilot

## Motivation

v2.11 identified that TF-IDF confidence measures vocabulary co-occurrence, not algorithmic
relevance. Three retrieval failure types were documented:

- **Type 1 — Surface lexical collision:** `merge_intervals` retrieves `merge_sorted` records
  (similarity 0.557) via shared tokens "merge" and "sorted."
- **Type 2 — Structural vocabulary overlap:** `tree_depth_tuple` retrieves `flatten` records
  via "nested", "list", "depth."
- **Type 3 — Repair vocabulary leak:** Repair records dense with family vocabulary score
  highly against all tasks in the family.

The oracle ceiling is 71.9% (23/32), with only 3 tasks recoverable by better routing.
Dense embeddings are the targeted response: they encode algorithmic structure rather than
token frequency.

## Retrieval Modes

### existing dense baseline
Code-aware MiniLM-L6 dense retrieval using local `code-memory-embedder`. The TF-IDF
fallback path in `memory/embed.py` is only active when this model is absent; it was
not active for any champion experiment.
k=4 optimal (per v2.6 ablation).

### dense
SentenceTransformer cosine similarity. Requires a pre-built dense index.
Default model: `sentence-transformers/all-MiniLM-L6-v2` (384d, CPU-compatible, ~80MB).
k=4 (same as TF-IDF for fair comparison).

### hybrid
Two-stage: TF-IDF selects top-N candidates (default N=20), dense model reranks to final top-k.
Preserves TF-IDF recall while applying semantic precision to the shortlist.

## Running the Experiment

### Step 1: Build dense index

```bash
conda run -n aetherforge-train python scripts/build_dense_memory_index.py \
    --source-index memory/index_adapted \
    --output-dir memory/dense_index_adapted \
    --dense-model sentence-transformers/all-MiniLM-L6-v2 \
    --batch-size 32 \
    --device auto
```

Or via Makefile: `make build-v217-dense-index`

### Step 2: Run evaluations

```bash
# Existing dense baseline (confirms reproducibility)
make eval-v217-tfidf-28
make eval-v217-tfidf-32

# Dense retrieval
make eval-v217-dense-28
make eval-v217-dense-32

# Hybrid retrieval
make eval-v217-hybrid-28
make eval-v217-hybrid-32
```

### Step 3: Summarise

```bash
make summarise-v217
```

## Promotion Rule

Dense or hybrid must beat the champion on the **clean 32-task benchmark**:
- +2 tasks (≥22/32 = 68.8%) = strong evidence → promote
- +1 task (21/32 = 65.6%) = weak evidence, within noise floor → do not promote
- 0 or negative → null result, champion retained

## Hardware Requirements

- CPU only: feasible for embedding 99 records; slow for model inference
- GPU recommended for evaluation (RTX 4080 Super or equivalent)
- Dense model: `all-MiniLM-L6-v2` requires ~500MB RAM, no GPU needed for indexing
