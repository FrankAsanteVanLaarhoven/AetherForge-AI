# AetherForge v2.10 — Clean Repair-Generalisation Benchmark

## Milestone goal

Test whether the v2.9 repair memory generalises beyond the known frozen benchmark failures,
using 32 untouched tasks across 5 pattern families — none overlapping with the frozen
28-task benchmark or the v2.9 5-task clean set.

## Research position entering v2.10

| Result | Value | Label |
|---|---|---|
| Clean champion (23/28) | 82.1% | Frozen baseline |
| Repair diagnostic (27/28) | 96.4% | Repair-index diagnostic |
| Corrected diagnostic (26/27) | 96.3% | Excl. spec-conflicted task |
| v2.9 clean transfer (4/5) | 80.0% | Early external signal |

The 80% v2.9 clean transfer signal is based on 5 tasks — too small to be conclusive.
v2.10 scales this to 32 tasks to establish whether the repair-memory approach
transfers robustly at scale.

## Benchmark design

32 tasks, no overlap with frozen 28-task benchmark or v2.9 clean set.

File: `data/v210_clean_repair_generalisation_tasks.jsonl`

| Family | N | Description |
|---|---|---|
| `interval_merge` | 6 | Interval union, insert, intersection, room scheduling, range summary, greedy removal |
| `sorted_selection` | 7 | Matrix kth, running median, k-way merge (heap), wiggle sort, smaller-after-self, peak finding, rotated search |
| `nested_dict` | 7 | Deep delete, flatten/unflatten (dotpath), deep merge, all keypaths, conditional update, recursive filter |
| `tuple_tree` | 7 | Leaf sum, leaf list, mirror, max-path sum, build-from-sorted, serialize/deserialize, width |
| `rle_encoding` | 5 | Roundtrip, generic list compress, expand, longest run, delta encode/decode |

## Evaluation lanes

```
Lane 1: memory/index_adapted     (99 records, clean champion baseline)
Lane 2: memory/index_adapted_v29 (103 records, repair-enhanced)
```

Both lanes use the same model (`outputs/qwen15b_v27_champion_merged`) and the
same eval settings (best-of-3, verified_agent, strict contract, k=4).

## Promotion rule

| Condition | Claim |
|---|---|
| Lane 2 >= 75% AND Lane 2 > Lane 1 by >= 5 pp | Repair memory generalises strongly — valid claim |
| Lane 2 >= 75% but NOT clearly > Lane 1 | Repair memory doesn't degrade clean tasks; champion already retrieves well |
| Lane 2 < 75% | Repair memory does not generalise broadly |
| Any family where Lane 1 > Lane 2 | Note regression — champion index retrieves better for that family |

## Per-family interpretation guide

| Family | What a Lane 2 win means |
|---|---|
| `interval_merge` | `merge_intervals` repair generalises to interval variants |
| `sorted_selection` | `median_two_sorted` repair generalises to sorted-array algorithms |
| `nested_dict` | `deep_get` repair generalises to related dict traversal tasks |
| `tuple_tree` | `tree_depth_tuple` repair generalises to tuple-tree structure tasks |
| `rle_encoding` | No repair record for RLE — any improvement is from existing memory |

## Claim boundaries

**Allowed after v2.10:**
- "v2.10 shows repair memory generalises to N/32 = X% of untouched tasks across 5 families."
- "Repair index beats champion index by Y pp on the 32-task clean benchmark."
- "Generalisation is strong in [family A, B] but limited in [family C]."

**Not allowed after v2.10:**
- Claiming the 96.4% diagnostic as a clean held-out score.
- Claiming production-grade reliability, SWE-bench success, or general superiority.
- Promotion of 23/28 clean champion based on v2.10 results (they use different tasks).
