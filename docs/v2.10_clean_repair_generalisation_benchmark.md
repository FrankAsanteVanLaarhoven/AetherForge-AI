# AetherForge v2.10 — Clean Repair-Generalisation Benchmark

## Milestone goal

Test whether the v2.9 repair memory generalises beyond the known frozen benchmark failures,
using 32 untouched tasks across 5 pattern families — none overlapping with the frozen
28-task benchmark or the v2.9 5-task clean set.

## Key findings

### Finding 1 — Repair records shuffle retrieval noise, not eliminate it

Adding 4 targeted repair records to the champion index changed retrieval for ALL 32 tasks,
not just the 4 repair targets. The repair records improved retrieval for 2 tasks
(`interval_union`, `running_median`) while degrading it for 4 others
(`deep_delete`, `unflatten_dict`, `tree_mirror`, `merge_sorted_k`, `rle_compress`).

This confirms the core diagnosis from v2.8/v2.9: **retrieval noise is a global index property,
not a per-task fix.** Adding records to the full index reshuffles similarity scores for all
tasks; the net effect is unpredictable.

### Finding 2 — The `interval_merge` repair generalises

`interval_union` flipped FAIL → PASS (repair fixed retrieval for this task).
`interval_merge` family improved from 4/6 → 5/6 (+16.7 pp).
This is a clean positive signal that the `merge_intervals` repair record helps
related interval tasks when retrieved correctly.

### Finding 3 — `nested_dict` and `tuple_tree` regress under repair index

The champion index already achieves 85.7% on nested_dict tasks (6/7).
The repair index injected the `deep_get` repair record context into tasks where it is
noise, causing `deep_delete` to regress (PASS → FAIL).

Similarly, `tree_mirror` regressed (PASS → FAIL) despite the tuple_tree repair record
targeting tree_depth, not mirror — the structural vocabulary overlap caused retrieval noise.

### Finding 4 — v2.9 clean generalisation (4/5 = 80%) was a sampling artefact

The v2.9 5-task clean set happened to contain tasks that benefited from repair retrieval.
The 32-task v2.10 benchmark reveals the full picture: the repair index causes as many
regressions as improvements on clean tasks. The 4/5 = 80% signal was not representative.

## Claim boundary

### What v2.10 allows

- **Champion baseline on 32 clean tasks:** 20/32 = 62.5% (Lane 1, memory/index_adapted).
- **Repair index on 32 clean tasks:** 18/32 = 56.2% (Lane 2, memory/index_adapted_v29).
- **Interval merge repair generalises:** +16.7 pp improvement with repair index on 6 interval tasks.
- **Retrieval noise is a global index property:** confirmed at scale.
- **Champion index (99 rec) is the superior retrieval configuration** for the current task distribution.

### What v2.10 does not allow

- Claiming repair memory generalises to unseen tasks broadly (it regresses overall).
- Claiming the 96.4% diagnostic as a clean score.
- Promoting `memory/index_adapted_v29` as the new champion index.

## Path forward

Two directions remain credible:

**Option A — Per-family subindex routing**
Route tasks to different subindexes based on task family classification.
`interval_merge` tasks → repair index; `nested_dict`, `rle` tasks → champion index only.
This requires a task classifier at inference time.

**Option B — Clean promotion (merge + re-eval)**
Merge the 4 repair records into `memory/raw_adapted/`, rebuild `memory/index_adapted`,
and re-run the 28-task frozen benchmark. If score ≥ 24/28, that is a new clean champion.
The merged index may or may not reproduce the diagnostic noise pattern depending on
how the merge changes the overall retrieval landscape.
