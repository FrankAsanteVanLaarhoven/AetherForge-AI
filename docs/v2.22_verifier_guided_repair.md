# v2.22 — Verifier-Guided Multi-Step Repair (Scoping)

Status: SCOPING (no results yet). This document defines the design, success criteria, and
evaluation protocol for the next milestone. No claims about outcomes are made here.

## Context

The v2.17–v2.21b arc resolved a three-way failure taxonomy on the clean 32-task benchmark
(baseline: code-aware MiniLM dense, mean 16.3/32):

| Failure type | Root cause | Evidence | Status |
|---|---|---|---|
| interval (e.g. `interval_union`) | coverage-bound | v2.19b/c: adding relevant verified memory converts it (0/3→3/3) | resolved |
| `tree_width` | reasoning/control-bound | v2.21 + v2.21b ablation: the execution-plan **structure** converts it (9/9), example-independent | resolved |
| `tree_serialize`, `tree_from_list`, `tree_max_path_sum` | capability-bound | v2.21/v2.21b: 100% plan adherence, still 0–1/6 | confirmed |

The three remaining failures are **capability-bound**: the model retrieves relevant memory
(v2.19c) and writes a correct high-level plan (v2.21b guard metrics: PLAN 100%, combine 100%),
but cannot produce the specific output logic — string serialization, balanced-BST
reconstruction, any-path dynamic programming. Neither more memory, a heavier embedder, nor more
planning has moved them.

The principle, stated across the arc, is **repair before fine-tuning**: before spending training
compute we test whether a bounded, verifier-guided repair loop — where execution feedback is
turned into a precise, actionable failure signal — lets the model self-correct these specific
execution errors. If it does not, the repair traces localize exactly where the model fails,
which is the right input to a future targeted fine-tune.

## Goal

For the three capability-bound tree tasks, test whether a **verifier-guided multi-step repair
loop** converts any of them, without modifying model weights.

Target conversion (the only promotion basis):
> At least one of `tree_serialize`, `tree_from_list`, `tree_max_path_sum` converts to a stable
> pass (3/3) across runs, without broad regression on the existing stable-pass tasks.

## Non-goals (do NOT do)

- No model-weight training/fine-tuning (that is a later milestone, informed by this one).
- No SWE-bench; no GGUF / 4-bit embedding backend; no new compiled CUDA dependency.
- No changes to protected memory indexes (`memory/index_adapted`, `index_adapted_v29`).
- No committing generated indexes, outputs, logs, checkpoints, or generated JSONL.
- No claim that repair will succeed; no performance numbers exist yet.

## Why the current loop is not already this

The existing agent loop already retries on error (forces `CRITIQUE:` after `OBSERVATION: ERROR`
or `(no output)`, then a corrected `TOOL_CALL`, under best-of-3). What it does **not** do:

1. The failure signal fed back is the raw subprocess stderr (an `AssertionError` traceback or
   exception). It is not distilled into *which* assertion failed, the *expected vs actual*
   value, and the *specific input* — the precise mismatch the model needs.
2. There is no explicit, bounded repair *budget* with a per-iteration "you must change X"
   contract; retries are opportunistic and may stall or repeat.
3. There is no measurement of repair dynamics (iterations used, whether the signal was
   actionable, whether each revision actually changed the failing behaviour).

v2.22 adds exactly these three things.

## Mechanism

### 1. Verifier (precise failure signal)

A deterministic post-execution analyzer that turns a failed `execute_code` observation into a
structured diagnostic, injected in place of the raw traceback:

```
VERIFIER:
  status: FAIL
  failing_check: assert tree_serialize((1,(2,3))) == "(1,(2,3))"
  expected: "(1,(2,3))"
  actual:   "1(23)"
  diff_hint: output is missing the parenthesis/separator structure
  input:    (1,(2,3))
```

Signal categories the verifier should distinguish (deterministically, by parsing stderr +
re-running the model's own asserts in a harness):
- `assertion_mismatch` — got expected/actual for the first failing assert.
- `exception` — exception type + the offending line (e.g. `IndexError` at line N).
- `no_output` — code ran but printed nothing / no PASS.
- `wrong_recursion_shape` (best-effort) — e.g. returned a scalar where a structure was expected.

The verifier must NOT reveal a reference solution. It reports only what the model's OWN code
did vs what the model's OWN asserts expected. The benchmark stays independent (the v210 tasks
use the generic PASS check; the model writes its own asserts, so the verifier surfaces the
model's self-test mismatch — no oracle leakage).

### 2. Bounded repair loop

- Repair budget: `--max-repair-iters` (default 3; sweep {2,3,4}).
- Each iteration after a FAIL: the model receives the VERIFIER block and must emit a
  `CRITIQUE:` naming the precise cause, then a **changed** `TOOL_CALL` (no-repeat enforced by
  hashing the prior code; an identical resubmission is rejected with a "must change" note).
- Terminate on first `OBSERVATION: PASS` (stop-after-pass) or budget exhaustion.
- Build on the v2.21 execution-plan prompt (PLAN → code → test → repair) so the model still
  plans first; v2.22 only sharpens the repair half.

### 3. Implementation sketch (source-only)

- `scripts/agent_loop.py`: a verifier formatter (deterministic; parses the executor result)
  and a repair-budget parameter in `run_agent` / `run_agent_best_of_n`; a
  `VERIFIER_REPAIR_SYSTEM` prompt (execution-plan contract + explicit repair contract).
- `scripts/evaluate_code_agent.py`: flags `--verifier-repair` and `--max-repair-iters N`
  (non-breaking; default off).
- `Makefile`: `eval-v222-repair-capbound-run{1,2,3}` (the 3 capability-bound tasks via
  `--task-ids`), `eval-v222-repair-32-run{1,2,3}` (full-32 control + regression check),
  optional `--max-repair-iters` sweep targets, `summarise-v222-repair`.
- `scripts/summarise_v222_repair.py`: per-task conversion vs v2.21b, repair-dynamics metrics,
  strict verdict; writes `results/v222_verifier_guided_repair/`.
- Retrieval unchanged: reuse the v2.19c expanded index (`memory/dense_index_v219c_confirm`).

## Metrics to report

Per capability-bound task and aggregate:
- pass count under repair vs v2.21b baseline (0–1/6).
- mean repair iterations used; % runs that hit the budget.
- verifier signal mix (assertion_mismatch / exception / no_output / wrong_shape).
- repair effectiveness: fraction of repairs that changed the failing behaviour (not just
  re-submitted), and fraction that converted FAIL→PASS within budget.
- full-32 aggregate and hard regressions (stable-pass → stable-fail).
- plan-adherence guard metrics (carried from v2.21).

## Evaluation protocol

- Capability-bound subset (`tree_serialize`, `tree_from_list`, `tree_max_path_sum`):
  `make eval-v222-repair-capbound-run{1,2,3}` (best-of-3 each; optionally 6 runs for stability).
- Full-32 control: `make eval-v222-repair-32-run{1,2,3}` (regression guard; not the promotion
  basis).
- Compare against:
  - v2.21b ablation (same tasks, plan prompt, no repair budget): the 3 tasks at 0–1/6.
  - baseline 16.3/32 and v2.21 18.3/32 for aggregate context only.

## Promotion rule (strict)

- Promotion candidate: ≥1 of the 3 capability-bound tasks converts to stable pass (3/3) with no
  hard regression.
- Strong: ≥2 convert across runs with no hard regression.
- Aggregate-mean movement is NOT a promotion basis if the capability-bound tasks do not move.
- A conversion counts only if the repair traces show the verifier signal preceded a behaviour
  change that produced the pass (i.e. the repair loop, not sampling variance, caused it).

## Claim boundary (strict)

- Proven so far (inputs to v2.22, not v2.22 claims): execution-plan structure converts
  `tree_width` example-independently; the other three tree tasks are capability-bound (fail
  despite 100% plan adherence).
- v2.22 will claim ONLY what the 32-task evidence supports.
- If repair converts ≥1 task: "verifier-guided multi-step repair converts a capability-bound
  tree task without fine-tuning" (bounded to these tasks, n=32, best-of-3).
- If repair fails: "bounded verifier-guided repair did not overcome the capability-bound tree
  failures; the repair traces localize the failure for a future targeted fine-tune."
- Not claimed: SWE-bench success; production reliability; model-weight improvement;
  frontier-model superiority; that repair removes the eventual need for fine-tuning.

## Relationship to fine-tuning

v2.22 is deliberately the step *before* training. Two outcomes both advance the program:
1. Repair converts tasks → a cheap, weight-free capability gain, and a clear spec for what a
   plan+repair agent can do.
2. Repair fails → the per-iteration repair traces are a precise, verified dataset of *where*
   the model's execution breaks, which is the correct, minimal input for a later targeted
   LoRA fine-tune (`v2.23`), rather than guessing.
