# v2.33 — Scaffold-First Tool-Call Preservation

Milestone type: **isolation pilot** (scaffold-only SFT, no repair). **Status: HOLD** — scaffold
dataset + GPU-gated trainer / eval / benchmark harness delivered and validated offline; the run is
**deferred to a GPU host** (this environment is CPU-only). No metrics are fabricated.

Branch: `feature/v233-scaffold-first-tool-call-preservation`.

## Scientific purpose

v2.33 isolates scaffold/tool-call preservation because **v2.31 and v2.32 both learned local repair
behaviour while collapsing execute_code/tool-use behaviour on the frozen benchmark**:

- v2.31 (repair-only): repair validation up, but the frozen 32-task agent benchmark collapsed to
  0–1/32 with `no_tool_call` dominant.
- v2.32 (repair + preservation): repair improved (base 0/10 → adapter 2/10), but tool-use
  preservation was 0/7, 32-task 0/32 vs champion 23, tree_serialize not preserved, `no_tool_call`
  still dominant.

Therefore v2.33 **does not add repair training**. It trains ONLY on correct tool-call scaffolds to
answer: *can we preserve execute_code/tool-use behaviour and the frozen 32-task benchmark before
reintroducing repair traces?* Success is **tool-call preservation without regression** — not repair
improvement.

## Phase 1 — Scaffold dataset (DONE, committed evidence)

`scripts/build_v233_scaffold_dataset.py` builds scaffold/tool-call preservation examples ONLY (no
repair objective, no failed-candidate examples): each is correct agent behaviour
`task → PLAN → TOOL_CALL: execute_code → VERIFIER PASS → FINAL_ANSWER`, from NON-held-out reference
functions (v2.30 + v2.26 tasks; names disjoint from the 32-task benchmark, any overlap rejected).

Result: **32** examples (all `tool_use_preservation`, **repair = 0**), train **25** / val **7**, 10
families (incl. `tree_serialize_repr`). Contamination violations: **0**. Output local-only:
`data/generated/v233/scaffold_{train,val}.jsonl`.

## Phase 2 — Scaffold-only training (GPU host)

`scripts/train_v233_scaffold_sft.py` — small fresh LoRA (r=8), short max length, low LR, few steps,
separate path `outputs/v233_scaffold_first_sft/`. Reuses the v2.32 `WeightedDataCollator` (strips
string metadata so the HF collator never tensorises non-tensor fields). GPU-gated; skips cleanly on
CPU. Champion + prior adapters never overwritten.

## Phase 3 — Evaluation (GPU host)

`scripts/eval_v233_scaffold_sft.py`: tool-use preservation validation, and `--benchmarks` (delegates
to `evaluate_code_agent.py --hf-lora <adapter>` for the frozen 32-task / hard-tree / tree_serialize
gate) with a **failure-reason breakdown** — `tool_call_rate`, `execute_code_rate`, `no_tool_call`
count and dominance. Repair is reported only as an optional diagnostic, never as a gate.

## Decision gate (scaffold-first)

`scripts/summarise_v233_scaffold_sft.py::decide` (pure, unit-tested). **PROMOTE** only if: stable
training; tool-use preservation ≥ 80%; frozen 32-task non-regression vs champion 23/28; tree_serialize
3/3 preserved; `no_tool_call` NOT dominant; contamination 0; protected artifacts unchanged.
**HOLD** if the benchmark is missing. **HOLD/REJECT** if `no_tool_call` remains dominant, preservation
< 80%, the 32-task regresses, or tree_serialize is not preserved.

Run end-to-end on a GPU host: `make v233-gpu-runbook`, then review
`results/v233_scaffold_first/summary.md`. Only after this passes should **v2.34** reintroduce repair
traces.

## Claim boundary

Claimed: v2.33 builds a scaffold-only tool-call preservation substrate + gated trainer/eval to test
whether tool-use and the frozen 32-task benchmark can be preserved before repair adaptation returns.
NOT claimed: repair improvement (the objective is intentionally absent), SWE-bench success, production
reliability, RL training, general SOTA, or frontier-level agent performance. No champion adapter or
memory index created/overwritten; scaffolds are correct trajectories on newly-authored non-held-out
tasks.
