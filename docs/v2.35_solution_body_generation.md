# v2.35 — Solution-Body Generation After Tool-Call Recovery

Milestone type: **inference-time measurement + strict verification** (NO training). **Decision:
HOLD/REJECT** per the strict gate — with a decisive diagnostic.

Branch: `feature/v235-solution-body-generation`.

## Scientific purpose

v2.35 follows v2.34 because **tool-call emission was largely recovered** (no_tool_call 26→1, tool-call
rate 0.19→0.97), **but benchmark success remained blocked at 5/32 by incorrect or incomplete
solution-body generation** — the emitted execute_code bodies were incomplete, weakly asserted, or
wrong. v2.35 targets the solution BODY, not tool-call emission and not repair training.

Core question: *once execute_code emission is recovered, can inference-time measurement/verification
raise the frozen 32-task score above 5/32 without fabricating code?*

## Components

- `scripts/solution_body_verifier.py` — pure, deterministic strict verifier. Verifies a body against
  each task's REAL benchmark assertions (parsed from the `Verify:` clause); the model's own
  `print('PASS')` is stripped and never trusted. Rejects: unconditional `print('PASS')` (fake pass),
  missing/model-invented asserts, empty/incomplete bodies (no required function definition), and code
  that only satisfies model-invented tests. Classifies each body: `strict_pass` / `fake_pass` /
  `incomplete_no_def` / `assertion_failure` / `no_benchmark_tests`.
- `scripts/eval_v235_solution_body_generation.py` — runs the frozen 32-task benchmark with the v2.34
  tool-call controller enabled and strict body verification, over the v2.33 transcripts. No new
  generation, no fabricated code, no injected solutions.
- `scripts/summarise_v235_solution_body_generation.py::decide` — the promotion gate.

## Result (deterministic, reproducible; frozen 32-task)

- **Strict-verified pass: 5/32** (v2.34 baseline 5; champion 23) — no improvement.
- tool-call rate **0.969**, no_tool_call **1** (v2.34 control retained).
- Body classification: `incomplete_no_def` **18**, `strict_pass` **5**, `assertion_failure` **7**,
  `fake_pass` **1 (rejected)**, `no_tool_call` **1**.
- Dominant failure: **`incomplete_no_def`**. tree_serialize **1/1** preserved.

## Decision: HOLD/REJECT

Gates: `tool_call_rate_retained` PASS, `no_tool_call_not_dominant` PASS, `fake_pass_rejected` PASS
(the 1 fake pass is classified and never counted), `tree_serialize_preserved` PASS, `artifact_safety`
PASS — but **`score_improves_over_5of32` FAIL** (5→5). Per the strict gate, **HOLD/REJECT**.

**Diagnostic (the value of v2.35):** the strict verifier confirms the 5/32 contains **no fake
passes**, and localises the residual bottleneck precisely — **18/32 bodies have no function definition
at all** (the model emits `TOOL_CALL:` + asserts without an implementation), and 7 more define a
function that fails the real assertions. Tool-call emission and format are no longer the problem; the
bottleneck is generating a correct implementation body, which an inference-time controller cannot
manufacture. That requires generation/training (v2.36+), not a controller.

## Claim boundary

v2.35 makes **no** claim of model improvement, repair improvement, SOTA, SWE-bench success, or
production readiness. No new generation, no training, no fabricated or injected solutions; no model,
champion adapter, or memory index was created or overwritten. Recovered/verified passes are checked
against the benchmark's real assertions only — so no fake pass can inflate the score.
