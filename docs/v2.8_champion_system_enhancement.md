# AetherForge v2.8 — Champion System Enhancement

## Milestone goal

Improve the 82.1% champion system (23/28 on the frozen 28-task benchmark)
to 24/28 (85.7%) or better, **without retraining the model**.

## Starting point

| System | Score |
|---|---|
| Champion adapter (unmerged) + memory | 22/28 = 78.6% |
| Merged champion + memory (current best) | 23/28 = 82.1% |
| Merged champion — no memory | 18/28 = 64.3% |

Champion path: `outputs/qwen15b_v27_champion_merged`
Memory index: `memory/index_adapted`

## Five failing tasks

| Task | Category | Hypothesis |
|---|---|---|
| group_anagrams | medium | grouping logic requires iteration not covered by retrieved examples |
| merge_intervals | hard | sort + merge pattern; memory provides wrong_retrieval or hurts |
| count_islands | hard | DFS/BFS grid traversal; no similar memory entry |
| median_two_sorted | hard | binary search on two sorted arrays; purely algorithmic gap |
| tree_depth_tuple | hard | tree traversal returning tuple; format constraint missed |

## Experiments

### 1. Reproduce baseline (`eval-v28-current-champion`)

Confirm 23/28 with current setup before any changes. Required first step.

### 2. Memory ablation (`eval-v28-no-memory`)

Run merged champion without memory. Confirms the 14.3 pp memory lift carries
over to the merged model and isolates the memory contribution.

### 3. Top-k ablation (`eval-v28-memory-topk1`, `topk3`, `topk5`)

The current system uses top-k = 4. The five failing tasks may be harmed by
retrieval noise (wrong examples pulled in) or helped by more context.

- **k = 1**: minimum noise; only the single closest example
- **k = 3**: moderate coverage
- **k = 5**: maximum context; risk of noisy examples outweighs benefit

### 4. Filtered memory (`eval-v28-filtered-memory`)

Build a new index from `memory/raw_adapted` but restricted to hard and medium
categories. Hypothesis: basic-category examples dilute retrieval quality for
hard tasks. Build the index before running:

```
python scripts/build_vector_memory.py \
    --raw-dir memory/raw_adapted \
    --index-dir memory/index_v28_filtered
```

### 5. Direct-answer prompt (`eval-v28-direct-answer-prompt`)

Replace `STRICT_SYSTEM` with `DIRECT_ANSWER_SYSTEM`, a more concise contract
that reduces scaffolding overhead. Hypothesis: for tasks like `median_two_sorted`
and `tree_depth_tuple`, the strict multi-step contract may cause the model to
over-structure its output at the cost of correctness.

Implemented via `--prompt-variant direct_answer` in `evaluate_code_agent.py`.

### 6. Continuation logic (`eval-v28-continuation-logic`)

Run best-of-5 sampling exclusively on the 5 failing tasks. Tests whether the
failures are due to sampling variance (solvable with more attempts) or a
fundamental capability gap (not solvable without retraining).

If best-of-5 passes all 5, the issue is sampling; if it still fails, the gap
is structural and requires a different intervention.

### Per-task findings

Tasks failing in the v2.8 baseline run: `rle_decode`, `deep_get`,
`merge_intervals`, `median_two_sorted`, `tree_depth_tuple`.

| Task | Across all 6 configs | Root cause |
|---|---|---|
| `median_two_sorted` | FAIL in all 6 | Structural gap — binary-search algorithm |
| `deep_get` | FAIL in all 6 | Structural gap — nested-key traversal edge cases |
| `rle_decode` | FAIL with all memory; PASS with direct-answer prompt | Prompt-sensitive; but direct-answer hurts 4 other tasks |
| `tree_depth_tuple` | PASS with no memory; FAIL with any retrieval | Any memory context misleads the tuple-return format |
| `merge_intervals` | FAIL at k=4 (baseline); PASS at k=1/k=5/filtered | Retrieval noise: 8 `merge_sorted` duplicates dominate k=4 |

### Targeted memory audit

Inspected k=4 retrieval for the two memory-sensitive tasks:
- `merge_intervals`: top-4 hits are 8 `merge_sorted` duplicates (score 0.5572).
  The word "merge" creates false similarity; the two-pointer merge pattern
  misleads the model away from the interval-merge algorithm.
- `tree_depth_tuple`: top-4 hits are `flatten` records (score 0.3902).
  Nested-list iteration pattern misleads the model into a flatten-style
  solution instead of recursive depth counting.

Filtered index (`memory/index_v28_filtered`, 83 records):
- Removed 8 `merge_sorted` records + 8 `flatten` records.
- `merge_intervals` PASSES in the filtered run (retrieval noise confirmed as cause).
- `tree_depth_tuple` still FAILS (BFS context is better but still not helpful).
- `matrix_transpose` FAILS in filtered run (sampling variance — passes in all
  other configs including no-memory; `flatten` context was coincidentally helpful).
- Net score: 23/28 = tie. Filtered index is not strictly better.

### What v2.8 established

1. **k=4 is the optimal setting.** k=1, k=3, k=5 all score lower.
2. **Memory is load-bearing.** Removing it drops 17.8 pp (64.3%).
3. **`merge_intervals` failure is retrieval noise**, not a model capability gap.
   It can pass when the noisy records are absent.
4. **`tree_depth_tuple` is memory-hurt.** Passes without any retrieval context;
   any retrieved example misleads the model.
5. **`median_two_sorted` and `deep_get` are structural gaps.** Fail in all 6
   configurations. Not addressable by memory or prompt tuning alone.
6. **Direct-answer prompt is net negative.** Fixes `rle_decode` but breaks
   `rotate_list`, `rle_encode`, `group_anagrams`, `parse_kv_lines` (−4 net).
7. **Continuation sampling (best-of-5)** resolves `group_anagrams`,
   `merge_intervals`, `count_islands` → these are sampling-variance fails, not
   structural. `median_two_sorted` and `tree_depth_tuple` still fail with 5
   attempts → confirmed structural.

### Next steps beyond v2.8

- The two confirmed structural gaps (`median_two_sorted`, `deep_get`) may be
  addressable by adding targeted verified examples to the memory index for those
  specific tasks (not retraining — just memory augmentation).
- `tree_depth_tuple` benefits from no-memory; a task-specific suppression rule
  could fix it without changing the index globally.
- These require index-level surgery, not model retraining.
