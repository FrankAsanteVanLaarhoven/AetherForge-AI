# AetherForge v2.8 — Champion System Enhancement

## Milestone goal

Improve the 82.1% champion system (23/28 on the frozen 28-task benchmark)
to 24/28 (85.7%) or better, **without retraining the model**.

## Starting point

| System | Score |
|---|---|
| Champion adapter (unmerged) + memory | 22/28 = 78.6% |
| Merged champion + memory (current best) | 23/28 = 82.1% |
| Merged champion — no memory | 18/28 = 64.3% |

Champion path: `outputs/qwen15b_v27_champion_merged`
Memory index: `memory/index_adapted`

## Five failing tasks

| Task | Category | Hypothesis |
|---|---|---|
| group_anagrams | medium | grouping logic requires iteration not covered by retrieved examples |
| merge_intervals | hard | sort + merge pattern; memory provides wrong_retrieval or hurts |
| count_islands | hard | DFS/BFS grid traversal; no similar memory entry |
| median_two_sorted | hard | binary search on two sorted arrays; purely algorithmic gap |
| tree_depth_tuple | hard | tree traversal returning tuple; format constraint missed |

## Experiments

### 1. Reproduce baseline (`eval-v28-current-champion`)

Confirm 23/28 with current setup before any changes. Required first step.

### 2. Memory ablation (`eval-v28-no-memory`)

Run merged champion without memory. Confirms the 14.3 pp memory lift carries
over to the merged model and isolates the memory contribution.

### 3. Top-k ablation (`eval-v28-memory-topk1`, `topk3`, `topk5`)

The current system uses top-k = 4. The five failing tasks may be harmed by
retrieval noise (wrong examples pulled in) or helped by more context.

- **k = 1**: minimum noise; only the single closest example
- **k = 3**: moderate coverage
- **k = 5**: maximum context; risk of noisy examples outweighs benefit

### 4. Filtered memory (`eval-v28-filtered-memory`)

Build a new index from `memory/raw_adapted` but restricted to hard and medium
categories. Hypothesis: basic-category examples dilute retrieval quality for
hard tasks. Build the index before running:

```
python scripts/build_vector_memory.py \
    --raw-dir memory/raw_adapted \
    --index-dir memory/index_v28_filtered
```

### 5. Direct-answer prompt (`eval-v28-direct-answer-prompt`)

Replace `STRICT_SYSTEM` with `DIRECT_ANSWER_SYSTEM`, a more concise contract
that reduces scaffolding overhead. Hypothesis: for tasks like `median_two_sorted`
and `tree_depth_tuple`, the strict multi-step contract may cause the model to
over-structure its output at the cost of correctness.

Implemented via `--prompt-variant direct_answer` in `evaluate_code_agent.py`.

### 6. Continuation logic (`eval-v28-continuation-logic`)

Run best-of-5 sampling exclusively on the 5 failing tasks. Tests whether the
failures are due to sampling variance (solvable with more attempts) or a
fundamental capability gap (not solvable without retraining).

If best-of-5 passes all 5, the issue is sampling; if it still fails, the gap
is structural and requires a different intervention.
