# v2.23 — Targeted Capability Adapter for Hard Tree Tasks (Scoping)

Status: SCOPING (no results yet). Defines design, contamination discipline, training setup,
evaluation, and promotion criteria. No outcome claims are made here.

## Context

The v2.17–v2.22b arc produced a fully ablation-attributed failure taxonomy on the clean
32-task benchmark (baseline mean 16.3/32):

| Failure type | Fixed by | Ablation evidence |
|---|---|---|
| interval (e.g. `interval_union`) | coverage (same-family memory) | v2.19c (interval_union 0→3/3) |
| `tree_width` | execution-plan structure | v2.21b (9/9, example-independent) |
| broad aggregate | structured VERIFIER signal | v2.22b (raw stderr 16.7 vs verifier 22.0) |
| `tree_serialize`, `tree_from_list`, `tree_max_path_sum` | **nothing yet** | survive all three above |

The three remaining tasks are **capability-bound**: they fail under coverage, planning, and
verifier-guided repair (v2.22: serialize 0/3, from_list 2/3, max_path_sum 1/3). They require a
weight-level intervention — a small targeted adapter.

## The non-negotiable discipline: do NOT poison the benchmark

The three hard tasks are **evaluation-only**. v2.23 must NOT train on their solutions, prompts,
tests, function names, or literal values. Instead:

1. Use the v2.22 repair traces ONLY as a **diagnostic** of the failure modes (what the model
   gets wrong: missing base case, wrong string format, wrong DP recurrence).
2. Author **same-family, different-task** verified training examples that exercise those
   capability patterns on DIFFERENT problems (mirroring the v2.19b/v2.19c/v2.21b discipline,
   which is why those results are credible).

This is the only way a positive v2.23 result is believable. If the hard tasks move after
training on *different* same-family tasks, that is genuine capability transfer.

## Goal / non-goals

Goal: a small targeted LoRA adapter that converts ≥1 capability-bound hard tree task to a
stable pass WITHOUT broad regression and WITHOUT dropping the full-32 below the v2.22 band.

Non-goals: no SWE-bench; no GGUF/4-bit embedding backend; no champion-adapter overwrite; no
training on held-out solutions; no claims of production reliability, frontier superiority, or
general SOTA.

## Heavy prior-warning (why this must be tightly controlled)

The project's record on retraining is uniformly negative (see
`memory/project_model_hierarchy.md`): v2.5 (53.6%), v2.6 trace-gating (best 57.1%), Option A
merge+fresh (64.3%), and LoRA-on-LoRA pilots all REGRESSED vs the frozen champion. The cause was
data-mixture / LR overwriting the champion's generalization. v2.23 therefore must:

- Train a SEPARATE adapter; never overwrite `outputs/qwen15b_memory_300steps/final` or the
  merged champion `outputs/qwen15b_v27_champion_merged`.
- Avoid LoRA-on-LoRA stacking; start from the same base the champion used, or from the merged
  champion explicitly, with the merge frozen.
- Use a SMALL, targeted dataset and few steps with early stop.
- Gate promotion on NO broad regression — a higher hard-tree score that damages stable families
  is a rejection, exactly as in v2.10/v2.11.

## Training-data construction

Source families (the capability patterns the hard tasks need), each as DIFFERENT tasks:
- tree serialization / deserialization (string/round-trip building) — NOT `tree_serialize`.
- tree reconstruction from a list / BST insert — NOT `tree_from_list`.
- tree any-path / max-path dynamic programming — NOT `tree_max_path_sum`.

Each verified record (committed source under `data/v223_tree_capability_records.jsonl`):

```json
{
  "task_family": "tree",
  "capability_gap": "string round-trip build from a recursive structure",
  "instruction": "<a DIFFERENT tree task prompt>",
  "retrieved_memory_summary": "<related verified pattern>",
  "execution_plan": ["assumptions", "base case", "combine logic", "minimal test", "repair"],
  "candidate_solution": "<initial>",
  "verifier_signal": {"status": "fail", "failed_case": "...", "expected": "...",
                      "observed": "...", "diagnosis": "..."},
  "repaired_solution": "<corrected, verified>",
  "minimal_tests": "<asserts>",
  "verified": true
}
```

Pilot size: 50–150 verified examples. Each `repaired_solution` must execute and pass its own
`minimal_tests` (reuse the `verify()` harness from `scripts/build_v219b_family_records.py`).

## Contamination guard (a committed, runnable check)

`scripts/check_v223_contamination.py` must assert, against the 32-task benchmark:

- 0 exact task-id overlap.
- 0 exact function-name overlap (vs the benchmark callables).
- 0 exact prompt overlap.
- 0 exact solution overlap.
- 0 exact minimal-test overlap (where practical).

The build must abort if any overlap is found. The three hard tasks remain evaluation-only.

## Training setup (controlled pilot)

- Base: the same Qwen2.5-Coder-1.5B base / merged champion the arc uses; do NOT stack on the
  active champion adapter.
- LoRA: small rank (e.g. 8–16), low LR (e.g. 1e-5–2e-5), 50–150 steps, early stop, warmup.
- Output: a NEW local-only path `outputs/qwen15b_v223_tree_capability_adapter/` (gitignored).
- Reuse `scripts/finetune_qwen_code_agent.py` patterns; agent-only, strict contract.
- No champion overwrite; no protected-index writes.

## Evaluation protocol

Evaluate the new adapter under v2.22 structured-verifier mode (the best inference config) and
the same v2.19c retrieval:
- Hard tree subset (`tree_serialize`, `tree_from_list`, `tree_max_path_sum`): 3 runs.
- Full 32-task benchmark: 3 runs.
- Reference cohorts (already available): champion + verifier (v2.22, mean 22.0); baseline 16.3.

Report: per-hard-task pass counts; full-32 mean/std; stable-fail→pass conversions;
stable-pass→fail regressions; whether the v2.22 aggregate band (~22) is preserved.

## Required ablation

To distinguish a weight-level capability gain from continued verifier dependence:
- `v2.23 adapter + structured verifier` vs `v2.23 adapter WITHOUT structured verifier`.
- If cheap: `champion + structured verifier` vs `v2.23 adapter + structured verifier`.

A genuine capability gain should show the adapter helping even with a weaker signal, or the
adapter beating the champion under the same verifier.

## Promotion rule (strict)

- Candidate: ≥1 hard tree task converts to stable pass (3/3) across runs, with NO broad
  regression and full-32 mean not below the v2.22 band.
- Strong: ≥2 hard tasks convert, OR 1 converts while full-32 stays ≥22.0.
- Reject if: hard tasks do not move; OR full-32 regresses materially; OR target-like tasks
  improve while previously-stable families regress (the v2.10/v2.11 failure mode).

## Claim boundary (strict)

- If it fails: "the targeted adapter did not overcome the residual capability-bound tree tasks
  under the contamination-guarded protocol."
- If 1 converts: "targeted same-family capability training is a candidate intervention for
  residual tree reasoning failures."
- If ≥2 convert: "the residual tree failures were partially weight-capability-bound and improved
  under contamination-guarded targeted adaptation."
- Not claimed: SWE-bench success; production reliability; general tree reasoning solved;
  frontier-model superiority; broad SOTA. The frozen champion (23/28 = 82.1%) is unchanged; a
  promoted v2.23 adapter is a separate, additive capability artifact, not a champion replacement.

## Why this is the right next step

Coverage, planning, and verifier-format have each been attributed by ablation. The residual
three tasks survive all of them, so the remaining lever is weight-level capability — but only a
contamination-guarded, separate-adapter, regression-gated pilot can test it credibly given the
project's uniformly negative retraining history. Train around the hard tasks, not on them.
