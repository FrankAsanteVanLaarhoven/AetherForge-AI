# v2.28 — Self-Improving Trace Dataset

Milestone type: **dataset infrastructure**. **Not a training milestone** — no model is trained, no
SWE-bench, no new model families, no GGUF, no new CUDA deps, no weight/champion/index changes.

Branch: `feature/v228-self-improving-trace-dataset` (based on `da111f7`, the v2.27 audit).

## Goal

Turn the local v2.26 / v2.27 trace reconstructions into one canonical, contamination-guarded,
quality-scored dataset suitable for LATER SFT, preference optimization, or lightweight scaffold
training.

## Sources (Phase 1, file evidence)

- `data/generated/v227/traces.jsonl` (36 records) — authoritative: genuine candidate/repaired
  transitions, computed contamination guard, repair fields.
- `data/generated/v226/traces.jsonl` (36 records) — earlier, weaker reconstruction of the SAME eval
  runs (candidate == final, declared-only prompt/solution guard); deduped to v2.27 by (task_id, run).

Both are local-only / gitignored. The dataset built here is also local-only.

## Canonical schema (Phase 2)

Each record: `record_id, source, task_id, task_family, capability_tag, representation, model_config,
prompt_mode, retrieved_memory, plan, candidate_solution, verifier_signal{status, failure_type,
expected, observed, diagnosis, repair_hint}, repair_plan, repaired_solution, final_solution,
final_status, quality{has_plan, has_verifier_signal, has_repair, candidate_differs_from_final,
repair_successful, format_control_used, quality_score}, contamination_guard{task/function/prompt/
solution/test overlap}, split, use_tags, rejection_reason`.

`capability_tag` ∈ {coverage, planning, verifier_format, scale, format_control, unknown}
(tree_serialize family → `format_control`).

## Tooling (Phase 3)

- `scripts/build_v228_self_improving_dataset.py` — normalises both sources into the schema,
  RECOMPUTES the contamination guard uniformly (fixing v2.26's declared-only overlap), ENRICHES the
  verifier signal by re-running the v2.27 format verifier (expected/observed/diagnosis/repair_hint),
  re-verifies the final by standalone execution, applies filters, and routes splits. Output is
  local-only: `data/generated/v228/dataset.jsonl` + `dataset_aggregate.json`.
- `scripts/summarise_v228_self_improving_dataset.py` — writes only the small committed summaries.
- `tests/test_v228_trace_schema.py` — schema + filter + aggregate-invariant tests.

## Quality filters (Phase 4)

Reject if: missing verifier signal; missing final status; claimed repair with no code change;
contamination overlap > 0; no plan; no task family; no representation; unverified final solution
(claims pass but does not execute); or superseded by a higher-priority source.

Training-use classification (non-exclusive `use_tags`): `sft`, `preference`, `format_repair`,
`verifier_format`; coarse `split` ∈ {train_candidate, preference_candidate, eval_only, rejected}.

## Result (Phase 5)

- Scanned **72** → accepted **29** / rejected **43** (36 v2.26 superseded; 7 degenerate
  claimed-repair-without-change).
- SFT candidates **26**; preference-pair candidates **7**; format-repair **0**;
  verifier-format **0**.
- Contamination guard violations: **0** (computed).
- Representation spread (accepted): exact_string 8, token_list 5, nested_list 7, json 9.

### Known limitations
- **Format-repair = 0 / verifier-format = 0**: the source runs' repair loop fixed 0 genuine failures
  (v2.27 `repair_attempted_fixed=0`); the 15 envelope-format failures were algorithm-correct, and
  degenerate repair records are filtered. So the substrate yields SFT positives and cross-record
  preference pairs but NOT broken→fixed repair pairs yet — a richer trace harvest is the next step.
- Bounded to the tree_serialize representation family (3B-bf16 runs). Substrate, not a broad dataset.
