# AetherForge v2.11 — Retrieval Routing and Gating Audit

## Context

v2.10 showed that globally adding repair records to the champion index
regresses clean generalisation (18/32 = 56.2% vs champion 20/32 = 62.5%).
The per-family breakdown revealed the mechanism: repair records shuffle
retrieval noise across all tasks, not just the 4 repair targets.

v2.11 tests whether **selective routing** can preserve repair gains for
the families that benefit (interval_merge) without causing regressions elsewhere.

## Research position entering v2.11

| Result | Value | Label |
|---|---|---|
| Clean champion (frozen 28-task) | 23/28 = 82.1% | FROZEN CHAMPION |
| Champion on 32 clean tasks | 20/32 = 62.5% | v2.10 clean baseline |
| Repair index on 32 clean tasks | 18/32 = 56.2% | v2.10 rejected |
| interval_merge repair gain | +16.7 pp | Only winning family in v2.10 |

## Results

| Family router (interval_merge → repair) | 19/32 = 59.4% | −3.1 pp | +3.2 pp |

| Oracle ceiling (perfect per-task routing) | 23/32 = 71.9% | +9.4 pp | +15.7 pp |

| Task | Family | Champion v2.10 | Repair v2.10 | Oracle decision |
|---|---|---|---|---|

### Confidence router failure analysis

The confidence router correctly identifies `running_median` and `deep_keys` as
repair-beneficial, but also routes entire families (nested_dict, tuple_tree) to
repair because the `deep_get` and `tree_depth` repair records score highest for
ALL tasks in those families (repair similarity 0.59–0.82).

Tasks routed to repair that regress:

| Task | Champion v2.10 | Repair v2.10 | Confidence route | v2.11 result |
|---|---|---|---|---|
| `v210_deep_delete` | PASS | FAIL | → repair | FAIL |
| `v210_unflatten_dict` | PASS | FAIL | → repair | FAIL |
| `v210_tree_mirror` | PASS | FAIL | → repair | FAIL |

Gains (+3) exactly cancel losses (−3) → tied with champion at 20/32.

## Key findings

### Finding 1 — Routing gains are within sampling variance

The predicted +1 task gain from family routing failed to materialise:
`interval_union` (the target improvement) is itself subject to sampling variance — it
passed in v2.10 repair, failed in v2.11 family-router sub-eval.
Family router got 19/32 = 59.4%, 1 below champion baseline.

Sampling variance across best-of-3 runs is approximately ±2–3 tasks on this benchmark.
Any routing scheme that gains fewer than 3 tasks over champion cannot be distinguished
from noise.

### Finding 2 — The confidence signal is measuring family vocabulary, not repair relevance

The TF-IDF similarity margin fires on **family-level vocabulary overlap**, not
task-specific repair relevance. The `deep_get` repair record scores 0.64–0.82
against ALL nested_dict tasks (including `deep_delete`, which repair hurts).
The `tree_depth` repair record scores 0.59–0.72 against ALL tuple_tree tasks.

This is the same retrieval noise problem that caused regressions in v2.10, now
appearing as false confidence signals. A reliable routing signal would need to be
algorithm-specific (e.g., "this task requires an interval merge pattern") rather
than vocabulary-level.

### Finding 3 — Oracle ceiling is 23/32 = 71.9% (+9.4 pp), not 32/32

Only 3 of 32 tasks flip from fail to pass with optimal repair routing.
9 tasks fail with BOTH indexes — they are not solvable by any routing scheme
given the current indexes. The maximum achievable gain from per-task routing
is bounded to these 3 tasks.

### Finding 4 — Confidence router is the best deployed strategy (tied champion)

Despite routing aggressively (20/32 to repair), the confidence router
achieves 20/32 = 62.5%, exactly matching champion. The gains and losses
cancel algebraically. This confirms that the confidence signal provides no
net benefit over the champion index for clean generalisation.

## Claim boundary

### Allowed after v2.11

- **Clean champion (frozen)**: 23/28 = 82.1% on 28-task benchmark (unchanged).
- **v2.10/v2.11 clean baseline**: 20/32 = 62.5% (champion index, stable across runs).
- **Oracle ceiling**: 23/32 = 71.9% (diagnostic; 3 tasks benefit from per-task routing).
- **Confidence router**: 20/32 = 62.5% (matches champion; no regression, no gain).
- "The TF-IDF similarity margin is not a reliable signal for repair vs champion routing."
- "Routing gains are within best-of-3 sampling variance (±2–3 tasks)."

### Not allowed after v2.11

- Claiming any routing scheme improves over champion on clean tasks.
- Claiming the 96.4% diagnostic as a clean held-out score.
- Claiming a new frozen 28-task champion without re-running the 28-task benchmark.

## Path forward

The research picture is now complete through v2.11. The findings stack:

| Version | Finding |
|---|---|
| v2.8 | Retrieval noise diagnosed: merge_sorted records harm merge_intervals; flatten records harm tree_depth |
| v2.9 | Repair memory fixes known failures diagnostically (27/28 = 96.4%); clean transfer 4/5 = 80% |
| v2.10 | Global repair-index rejected: regresses clean generalisation (18/32 vs 20/32 champion) |
| v2.11 | Routing does not reliably recover repair gains; oracle ceiling is 23/32 = 71.9% |

**Conclusion:** The retrieval system's TF-IDF vocabulary similarity cannot distinguish
between "this task needs this repair pattern" and "this task shares vocabulary with this
repair pattern." The gap between oracle (23/32) and champion (20/32) is exactly 3 tasks,
all requiring algorithm-specific rather than vocabulary-level routing signals.

Two credible continuation paths:

**Option A — Dense retrieval (semantic embeddings)**
Replace TF-IDF similarity with a code-specific semantic embedder (e.g., CodeBERT,
UniXcoder). Semantic similarity would better distinguish algorithm patterns from
vocabulary families, reducing false-positive repair routing.

**Option B — Manuscript and SWE-bench**
The research story is publishable as-is. Five controlled experiments with clear
causal findings. The champion (23/28 = 82.1%) is the stable result.
Focus shifts to SWE-bench infrastructure and paper writing.
