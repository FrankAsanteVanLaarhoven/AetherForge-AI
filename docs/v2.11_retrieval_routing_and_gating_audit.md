# AetherForge v2.11 — Retrieval Routing and Gating Audit

## Context

v2.10 showed that globally adding repair records to the champion index
regresses clean generalisation (18/32 = 56.2% vs champion 20/32 = 62.5%).
The per-family breakdown revealed the mechanism: repair records shuffle
retrieval noise across all tasks, not just the 4 repair targets.

v2.11 tests whether **selective routing** can preserve repair gains for
the families that benefit (interval_merge) without causing regressions elsewhere.

## Research position entering v2.11

| Result | Value | Label |
|---|---|---|
| Clean champion (frozen 28-task) | 23/28 = 82.1% | FROZEN CHAMPION |
| Champion on 32 clean tasks | 20/32 = 62.5% | v2.10 clean baseline |
| Repair index on 32 clean tasks | 18/32 = 56.2% | v2.10 rejected |
| interval_merge repair gain | +16.7 pp | Only winning family |

## Routing strategies

All routers use the same 32 v2.10 clean tasks and the same model
(`outputs/qwen15b_v27_champion_merged`). Only the memory index selection changes.

### Lane 1 — Family router

Use repair index only for `interval_merge` tasks (6/32).
Use champion index for all other families (26/32).

**Theoretical upper bound if perfect family routing works:**
```
interval_merge:   5/6 = 83.3% (repair)
sorted_selection: 3/7 = 42.9% (champion)
nested_dict:      6/7 = 85.7% (champion)
tuple_tree:       3/7 = 42.9% (champion)
rle_encoding:     4/5 = 80.0% (champion)
Total: 21/32 = 65.6%
```
Maximum possible gain from family routing: +3.1 pp over champion baseline.

### Lane 2 — Confidence router

Default to champion index.
Route to repair index only if:
- Repair top-1 retrieval score >= 0.35 (reasonably relevant match)
- Repair top-1 score exceeds champion top-1 by >= 0.05 (repair clearly better)

### Lane 3 — Oracle router (diagnostic ceiling)

Per-task: use the index that passed in v2.10.
If repair passed and champion failed → oracle uses repair result.
Otherwise → oracle uses champion result.

This is the theoretical ceiling for any routing scheme — not deployable.

## Promotion rule

| Outcome | Interpretation |
|---|---|
| Router beats champion by >= 5 pp | Strong evidence for retrieval gating as a strategy |
| Router beats champion by < 5 pp | Marginal improvement; routing is correct direction but insufficient |
| Router matches champion | Routing preserves champion performance; no regression |
| Router regresses vs champion | Routing signal is too noisy; family/score signals are inadequate |
| Oracle >> deployed routers | Index selection is the bottleneck; need better routing signals |

## Claim boundaries

### Allowed after v2.11

- "Family router achieves N/32 = X% on 32 clean tasks."
- "Confidence router achieves N/32 = X% on 32 clean tasks."
- "Oracle routing ceiling is N/32 = X% (diagnostic, not deployable)."
- If router >= 68% (champion + 5 pp): "Selective retrieval routing is effective."

### Not allowed after v2.11

- Claiming any router as the new frozen 28-task champion.
- Claiming the 96.4% diagnostic as a clean held-out score.
- Claiming production-grade reliability.

### If routing is effective (>= 65%): next steps

1. Apply family router at frozen 28-task benchmark to measure clean champion impact.
2. Extend family classification to support all 5 families.
3. Consider building per-family subindexes rather than routing between two full indexes.

### If routing is not effective (< 62.5%): next steps

1. Routing is the wrong abstraction — the index boundary is too coarse.
2. Consider query-rewriting or direct retrieval-score filtering.
3. The 23/28 = 82.1% champion is the stable result; focus on manuscript and SWE-bench.
