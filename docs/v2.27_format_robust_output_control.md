# v2.27 — Format-Robust Output Control

Milestone type: **evaluation + infrastructure** (trace-quality, format-verifier, format-control).
**Not a training milestone** — no SFT, no preference optimization, no adapter, no model-weight
change. Training is explicitly deferred (see §9) until the trace factory produces reliable
broken→fixed repair pairs.

Branch: `feature/v227-format-robust-output-control` (based on `be6d070`, the v2.26 audit).

## 0. Grounding (from v2.26, file evidence)

- v2.26 held the algorithm constant and varied only output FORMAT for `tree_serialize`. At
  3B-bf16 + structured verifier (clean config), per-format pass rate
  (`results/v226_self_improving_traces/by_representation.csv`): exact_string 3/9 (33%),
  token_list 7/9 (78%), nested_list 4/9 (44%), json 2/9 (22%). Verdict **FORMAT_SENSITIVE**.
- The v2.26 trace factory had a critical gap: `candidate_solution == final_solution`, so it did
  not produce genuine repair transitions. Fixing that is the v2.27 priority.

## 1. Goal

Characterise the residual `tree_serialize` difficulty as a format-control problem and build the
infrastructure to act on it, without training: (a) harden the trace factory so it records genuine
repair transitions with a computed contamination guard; (b) add a structured tree-serialization
format verifier; (c) test whether a canonical intermediate representation (IR) + structured format
verification yields stable `tree_serialize` conversion where the raw model could not.

## 2. Hypothesis (tested)

H1: A canonical-IR + structured-format-verification control layer converts `tree_serialize`
stably across output formats, and the model's own failures are dominated by output-FORMAT control
(not algorithm). **Supported** (see §8).

## 3. Protocol (as implemented — deterministic, model-free)

All arms are deterministic and require no GPU and no new model run. The model baseline is mined
from the **existing** v2.26 transcripts (`outputs/eval_v226_3b_run{1,2,3}/best_of_3.csv`).

- **Phase 1 — trace factory hardening** (`scripts/build_v227_trace_factory.py`). Parse the genuine
  repair transitions already present in the v2.26 transcripts (PLAN → TOOL_CALL → VERIFIER →
  CRITIQUE → CHANGED_TOOL_CALL). Each trace records a DISTINCT first candidate and final/repaired
  solution, the structured verifier signal, the repair plan, and a RECOMPUTED ground-truth status
  (extracted code executed standalone) — independent of the unreliable eval CSV repair flags.
- **Phase 2 — format verifier** (`scripts/v227_format_verifier.py`). A canonical IR (nested-tuple
  tree) with correct-by-construction renderers, plus a classifier that labels a discrepancy as one
  of the seven failure types and emits a structured block (separate from raw stderr).
- **Phase 3 — format-control evaluation** (`scripts/v227_format_control_eval.py`). Arm A: model
  baseline per format (mined). Arm B: canonical-control render+verify over a deterministic 40-tree
  battery (seed 227) for all 3 logical tasks × 4 formats. Arm C: inject each of the 7 fault
  classes, confirm the verifier assigns the correct label, then apply canonical re-render and
  confirm conversion.

## 4. Format modes

The four v2.26 representations are reused verbatim — `exact_string`, `token_list`, `nested_list`,
`json` — across 3 logical serializations (`full_structure`, `leaf_values`, `leaf_depth`). The
controlled output modes evaluated are: free-form/exact-string, token-list, tuple/list,
JSON-like tree, and the **post-processed canonical string** (canonical IR re-render).

## 5. Format verifier

Failure types: `algorithmic_error`, `format_error`, `missing_null_marker`, `extra_null_marker`,
`separator_error`, `ordering_error`, `type_error`. Structured block:
`{status, failure_type, expected_format, observed_format, diagnosis, repair_hint}`. "null marker"
generalises to the format's structural delimiter (lisp parens, list brackets, json wrapper keys).

## 6. Repair loop

The hardened trace distinguishes `no_repair_needed` / `repair_attempted_failed` /
`repair_attempted_fixed`, and `repair_kind` ∈ `{none, format_only, algorithmic, envelope_format}`.
`envelope_format` marks traces whose extracted code is algorithmically correct standalone but was
scored FAIL because the tool-call/output ENVELOPE format was invalid (e.g. triple-quoted code
breaking JSON args) — direct evidence that the failure is format-control, not algorithm.

## 7. Contamination guard

`scripts/build_v227_trace_factory.py` COMPUTES overlap (not declared) against the 32-task
benchmark and the v2.26 slice for: task-name, function-name, prompt, solution, and test-case
overlap, and fails closed if any violation is found. Result: 0 violations.

## 8. Success gate & result

Gate: stable 3/3 conversion of the 3 logical `tree_serialize` tasks in the hardest format
(`exact_string`) via canonical control across the full battery — best-of-3 noise does not count —
with no broad regression.

- Canonical control: **3/3 stable** on `exact_string`; **100%** across all 12 (logical×format)
  cells (40 trees each). PASS.
- Verifier: each of the 7 fault classes classified with the correct label and canonically repaired
  (≥39/40 per class; the lone miss is a single-leaf no-op).
- Model contrast (Phase 1, 36 traces): repair loop fixed **0** genuine failures; **15/36** were
  envelope/format failures (algorithm correct, agent scored fail); only **7/36** were genuine
  candidate≠final transitions (the model usually resubmitted identical code).
- No model weights changed ⇒ no benchmark regression is possible by construction; the frozen 1.5B
  champion (23/28 = 82.1%) and all memory indexes are untouched.

Verdict: **FORMAT_CONTROL_RESOLVES** — the residual `tree_serialize` difficulty is
output-format/control-bound and is resolved by an inference-time format-control layer, not by
changing the model. Evidence: `results/v227_format_robust/{summary,claim_boundary}.md`,
`format_control.csv`.

## 9. Failure interpretation & deferred training

- The model's own verifier-repair loop converting 0/8 genuine failures, plus only 7/36 genuine
  transitions, means the existing traces are too thin in successful repair pairs to train on. SFT /
  preference optimization is therefore DEFERRED until the trace factory (or a future eval run)
  yields a sufficient corpus of genuine broken→fixed pairs. v2.27 deliberately builds and validates
  that infrastructure first.

## 10. Claim boundary

- Inference-time format-control result only; no training, no model-weight change. Bounded to the
  `tree_serialize` family and these formats. No SWE-bench, production-reliability, frontier, or
  broad-SOTA claim. The canonical renderers are correct by construction; the measured claim is that
  FORMAT CONTROL (not algorithm) is the bottleneck — supported by the 15/36 envelope-format
  failures and the model's 0 successful genuine repairs.
