# v2.31 — Tiny Repair-Trace SFT Pilot

Milestone type: **first training milestone** (tiny supervised fine-tuning pilot).
**Status: HOLD** — dataset export + GPU-gated trainer/eval harness delivered and validated offline;
the actual training/eval run is **deferred to a GPU host** (this environment is CPU-only).

Branch: `feature/v231-tiny-repair-trace-sft` (based on pushed v2.30).

## Goal

Test whether the verifier-labelled repair traces can train local repair behaviour: given task +
failed candidate + structured verifier signal, produce repair plan + corrected solution. Objective is
STABILITY, not SOTA.

## Phase 1 — SFT dataset export (DONE, committed evidence)

`scripts/build_v231_repair_sft_dataset.py` exports a LOCAL-ONLY SFT dataset from the genuine repair
traces (v2.29 + v2.30). Each example:

```
INPUT : task instruction + failed candidate + structured verifier signal + failure_type + repair_hint
OUTPUT: repair plan + corrected solution
```

Only contamination-clean records (guard all-zero, function/task names disjoint from the 32-task
benchmark) are kept; held-out benchmark solutions are never included.

- Total available: **50** | train **40** / validation **10** (deterministic every-5th hold-out).
- Format-repair **28** | algorithmic-repair **22** | **10 task families**.
- Sources: v2.29 9, v2.30 41. Contamination violations: **0**. Rejections: none.

Output (local-only / gitignored): `data/generated/v231/sft_{train,val}.jsonl`, `sft_aggregate.json`.

## Phase 2 — Tiny SFT run (NOT RUN — GPU-gated)

`scripts/train_v231_repair_sft.py`: a fresh small LoRA adapter (r=8, α=16, target q/k/v/o), short
max length (512), low LR (1e-4), few steps (60, early-stop friendly), bf16, **separate output path**
`outputs/v231_tiny_repair_trace_sft/` — the frozen champion and prior 3B/7B adapters are never loaded
for writing or overwritten. The script is GPU-gated: in this CPU-only environment
(`torch 2.x+cpu`, no CUDA) it prints a SKIP notice and exits 0. **No training metrics are fabricated.**
Preferred order on a GPU host: 1.5B smoke first; 3B if memory allows; 7B only if specifically approved.

## Phase 3 — Evaluation (NOT RUN — GPU-gated)

`scripts/eval_v231_repair_sft.py` evaluates, on a GPU host: (1) the v2.30 repair validation slice
(generate corrected solution → execute → pass rate), and delegates (2) the frozen 32-task benchmark,
(3) the hard tree subset, and (4) the tree_serialize 3/3 format-control check to the existing
`evaluate_code_agent.py` make targets (so the champion eval is reused, not perturbed). Comparisons:
base+verifier vs adapter+verifier vs adapter without verifier.

## Decision

**HOLD.** The export runs and is committed; the trainer and eval harness are import-safe and skip
cleanly with no fabricated metrics. Promotion requires a stable training run and a non-regressing
32-task benchmark, which a CPU-only environment cannot produce. Resume on a GPU host:

```bash
make build-v231-sft-dataset   # (done here) local-only export
make train-v231               # GPU host: tiny LoRA adapter -> outputs/v231_tiny_repair_trace_sft/
make eval-v231                # GPU host: repair-validation + delegated benchmark evals
make summarise-v231           # fills in training/eval metrics when present
```

Promotion (on a GPU host) requires: training completes without instability; contamination guard
remains 0; adapter improves/preserves repair validation; 32-task benchmark does not materially
regress; tree_serialize 3/3 preserved; no protected artifact overwritten.

## GPU runbook (run end-to-end on a GPU host)

`scripts/run_v231_gpu_runbook.sh` (or `make v231-gpu-runbook`) runs the full pilot on a GPU host and
refuses (exit 1, nothing trained) without a CUDA device — no metrics are fabricated. It:

1. **GPU gate** — verifies `torch.cuda.is_available()`; refuses otherwise.
2. **Protected-asset guard** — snapshots a sha256 manifest of the champion
   (`outputs/qwen15b_memory_300steps/final`) and the memory indexes (`memory/index_adapted*`,
   `memory/dense_index_v2*`) before and after the run and aborts if anything changed; refuses if the
   output path overlaps any protected path.
3. **Pipeline** — `build-v231-sft-dataset` → `train_v231_repair_sft.py --base
   Qwen/Qwen2.5-Coder-1.5B-Instruct --max-steps 60` (fresh LoRA, separate path
   `outputs/v231_tiny_repair_trace_sft/`) → `eval_v231_repair_sft.py` → `summarise-v231`.
4. **Promotion gates** — prints the checklist below; the decision is made by a human reading
   `results/v231_tiny_repair_trace_sft/summary.md`.

```bash
# on a GPU host, from the repo root:
make v231-gpu-runbook
# or, choosing base / steps explicitly:
bash scripts/run_v231_gpu_runbook.sh Qwen/Qwen2.5-Coder-1.5B-Instruct 60
```

All training/eval artifacts stay local-only (`data/generated/v231/`,
`outputs/v231_tiny_repair_trace_sft/`, gitignored); commit only the small
`results/v231_tiny_repair_trace_sft/*` summary. Escalate to a 3B base only if memory allows; 7B only
on explicit approval.

**Promotion requires (all must hold):** training completes without instability; contamination guard
remains 0; repair validation improves OR preserves high repair performance (adapter ≥ base); the
frozen 32-task benchmark does not materially regress vs champion 23/28; tree_serialize 3/3
format-control is preserved; no protected champion adapter or memory index is overwritten.

## Claim boundary

Claimed (when run): v2.31 provides an initial supervised repair-trace adaptation pilot showing whether
verifier-labelled repair traces can train local repair behaviour. NOT claimed: SWE-bench success,
production reliability, RL training, general SOTA, frontier-level agent performance, broad model
improvement. No weights, champion adapters, or memory indexes were created or overwritten in this
milestone.
