# v2.32 — Tool-Use Preservation During Repair-Trace Adaptation

Milestone type: **training pilot** (mixed-objective SFT). **Status: HOLD** — mixed dataset + GPU-gated
split-loss trainer / eval / benchmark harness delivered and validated offline; the training/eval run
is **deferred to a GPU host** (this environment is CPU-only). No metrics are fabricated.

Branch: `feature/v232-tool-use-preservation` (based on v2.31).

## Goal

v2.31 trains repair behaviour but risks eroding the tool-call / scaffold behaviour the frozen 32-task
benchmark depends on. v2.32 mixes **repair traces** with **tool-use / scaffold preservation traces**
under a **split loss** (repair objective + tool-use preservation objective), targeting: repair
validation **improves** *without dropping* the 32-task benchmark.

## Phase 1 — Mixed dataset (DONE, committed evidence)

`scripts/build_v232_mixed_dataset.py`:
- **repair examples** reuse the v2.31 export (failed candidate + verifier signal → repair plan +
  corrected solution), `objective="repair"`, `loss_weight=1.0`.
- **tool-use preservation examples** are correct agentic tool-call trajectories
  (`PLAN → execute_code → VERIFIER PASS → FINAL_ANSWER`) built from NON-held-out reference functions
  (v2.30 tasks + v2.26 representation tasks), `objective="tool_use_preservation"`, configurable
  `loss_weight`.

Result: **82** examples (repair **50** + preservation **32**, mix **0.61 / 0.39**), train **65** /
val **17** (val = 10 repair + 7 preservation), preservation across **10 families** (incl.
`tree_serialize_repr` ×12). Contamination violations: **0** (function names disjoint from the
benchmark; any overlap rejected). Output local-only: `data/generated/v232/sft_{train,val}.jsonl`.

## Phase 2 — Split-loss training (GPU host)

`scripts/train_v232_mixed_sft.py` — small fresh LoRA (r=8), short max length (512), low LR, few steps
(80), **separate path** `outputs/v232_tool_use_preservation_sft/`. A `SplitLossTrainer` scales each
example's LM loss by its objective `loss_weight` (repair vs preservation). GPU-gated; skips cleanly on
CPU. Champion and prior adapters never loaded for writing or overwritten.

## Phase 3 — Evaluation (GPU host)

`scripts/eval_v232_mixed_sft.py`: repair validation (base vs adapter), tool-use preservation
validation (output keeps the `execute_code` scaffold AND the final solution passes), and
`--benchmarks` (delegates to `evaluate_code_agent.py --hf-lora <adapter>` for the frozen 32-task /
hard-tree / tree_serialize gate).

## Decision (gate: repair improves without dropping the 32-task benchmark)

`scripts/summarise_v232_mixed_sft.py::decide` is three-tier; **the benchmark gate is mandatory for
PROMOTE**:

- **HOLD** — not trained.
- **PARTIAL PROMOTE / HOLD PENDING BENCHMARKS** — training + repair-improvement + tool-use
  preservation + artifact gates pass but the 32-task benchmark has not been run.
- **PROMOTE** — only when **all** pass: stable training; repair validation **improves** (adapter >
  base); tool-use preserved (≥ 80%); 32-task non-regression vs champion 23/28; tree_serialize 3/3
  preserved; contamination 0; no protected artifact overwritten.

Run end-to-end on a GPU host: `make v232-gpu-runbook` (or
`bash scripts/run_v232_gpu_runbook.sh <base> <max_steps>`), then review
`results/v232_tool_use_preservation/summary.md`.

## Claim boundary

Claimed: v2.32 builds a mixed repair + tool-use-preservation substrate with a split-loss trainer and a
gated evaluation to test whether tool-use can be preserved while adapting on repair traces. NOT
claimed: SWE-bench success, production reliability, RL training, general SOTA, frontier-level agent
performance, broad model improvement. No champion adapter or memory index created/overwritten;
preservation traces are correct scaffolds on newly-authored non-held-out tasks.
