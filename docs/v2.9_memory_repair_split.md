# AetherForge v2.9 — Memory Repair Split

## Milestone goal

Diagnose and repair the retrieval noise causing the 4 structurally failing tasks,
using a split architecture that preserves the champion index and prevents
benchmark contamination.

## Starting point

| System | Score |
|---|---|
| Merged champion + memory (23/28) | 82.1% = current best |
| Promotion threshold | 24/28 = 85.7% |

Failing tasks after v2.8 analysis:

| Task | v2.8 finding |
|---|---|
| `merge_intervals` | Retrieval noise — `merge_sorted` records dominate at k=4 (score 0.5572) |
| `median_two_sorted` | Retrieval noise — same `merge_sorted` records at even higher score (0.6399) |
| `deep_get` | Retrieval noise — `invert_dict` + `flatten` records (0.3958 / 0.3656) |
| `tree_depth_tuple` | Retrieval noise — `flatten` records at k=4 (score 0.3902) |

All four failures share a common pattern: the k=4 retrieval returns wrong-pattern
examples that mislead the model at inference time. None are model capability gaps —
removing or correcting the retrieved context should help.

## Architecture

```
memory/index_adapted          ← PROTECTED champion index (99 records, unchanged)
memory/raw_v29_repair/        ← 4 new verified repair records (separate)
memory/index_v29_repair/      ← champion + repair = 103 records (diagnostic use only)
data/v29_clean_generalisation_tasks.jsonl   ← 5 untouched tasks for clean eval
```

The champion index is never modified in v2.9. All repair records live in a
separate raw directory. The repair index is built by combining both for
diagnostic evaluation only.

## Retrieval analysis

### merge_intervals

Top-4 champion hits: all `merge_sorted` records (score ≈ 0.5572, all identical).
The word "merge" in the task creates false TF-IDF similarity to `merge_sorted` records.
The two-pointer merge pattern is not transferable to interval merging.

### median_two_sorted

Top-4 champion hits: all `merge_sorted` records (score ≈ 0.6399, **higher** than
`merge_intervals`). The phrase "two sorted" strongly matches `merge_sorted`
query_text ("merge two sorted lists"). The model receives confident wrong context
for this task — the highest-confidence wrong retrieval of any failing task.

### deep_get

Top-4 champion hits: 1× `invert_dict` (score 0.3958), 3× `flatten` (score 0.3656).
`invert_dict` shares dict-operation vocabulary; `flatten` shares nested-structure
vocabulary. Neither is relevant to nested-key path traversal with default fallback.

### tree_depth_tuple

Top-4 champion hits: all `flatten` records (score ≈ 0.3902).
The "nested" vocabulary in both tasks creates false similarity.
Flatten's list-iteration pattern misleads the model away from recursive
depth counting.

**Additional finding:** The `tree_depth_tuple` task prompt contains a broken assertion:
```
tree_depth(((1,2),(3,(4,5)))) == 3   # WRONG — correct value is 4
```
By the stated rule (leaves are depth 1, branch = 1+max(children)):
- `tree_depth((4,5))` = 2
- `tree_depth((3,(4,5)))` = 3
- `tree_depth(((1,2),(3,(4,5))))` = **4**, not 3

Any correct implementation returns 4. If the model copies the task's assertion
verbatim, the assert fails. If the model uses the correct value (4) in its test
code, the assert passes. Repair memory explicitly documents the correct value.
The pass/fail result for `tree_depth_tuple` may depend on whether the model
follows the task prompt or the repair memory example.

## Repair records

File: `memory/raw_v29_repair/repair_records.jsonl` (4 records)

| Task | Approach in repair record |
|---|---|
| `merge_intervals` | Sort by start, sweep with `merged[-1][1]` extension |
| `median_two_sorted` | O(m+n) linear merge, select middle element(s) as float |
| `deep_get` | Iterate keys, check `isinstance(d, dict) and key in d`, return default |
| `tree_depth_tuple` | `isinstance(node, int)` → 1, else `1 + max(recurse(l), recurse(r))` — uses correct assertions (==4 for the broken example) |

All 4 records pass schema validation (`memory/validate.py`) and code execution.

## Experiments

### 1. Retrieval inspection (`inspect-v29-retrieval`)

Run both champion and repair indexes for all 4 failing tasks at k=4.
Label each hit HELPFUL / HARMFUL / NEUTRAL.
Expected: champion index shows all HARMFUL; repair index shows HELPFUL at rank 1.
Writes: `results/v29_memory_repair/retrieval_inspection.md`

### 2. Build repair index (`build-v29-repair-memory`)

Combine `memory/raw_adapted` + `memory/raw_v29_repair` → `memory/index_v29_repair`.
Champion index (`memory/index_adapted`) is not touched.

### 3. Diagnostic repair eval (`eval-v29-repair-memory-diagnostic`)

Run merged champion + repair index on the original 28-task frozen benchmark.
**Label: DIAGNOSTIC — not a clean result.**
Rationale: the index now contains task-specific repair examples for the exact
tasks being evaluated. Any improvement is contaminated by this overlap.
Use to confirm the repair examples actually help the model, not to claim promotion.

### 4. Clean generalisation eval (`eval-v29-clean-memory-generalisation`)

Run merged champion + repair index on `data/v29_clean_generalisation_tasks.jsonl`.
5 tasks similar to the failing tasks but with different names, examples, and variants:
- `v29_clean_merge_time_slots` → same sort-and-sweep as `merge_intervals`
- `v29_clean_find_kth_sorted` → same two-sorted-merge as `median_two_sorted`
- `v29_clean_deep_set` → same nested-key traversal as `deep_get` (write, not read)
- `v29_clean_tree_node_count` → same tuple-tree recursion as `tree_depth_tuple`
- `v29_clean_merge_k_sorted` → k-list generalisation of `median_two_sorted`'s merge

**Label: CLEAN** — valid generalisation claim if score is high.
These tasks were not used during training, memory construction, or v2.8 evaluation.

## Promotion rules

| Scenario | Claim allowed |
|---|---|
| Diagnostic score >= 24/28 | Repair examples reduce known failures (diagnostic) — NOT a new champion |
| Diagnostic score < 24/28 | Repair examples do not close the gap on known failures |
| Clean score >= 75% (3/4+) | Repair memory **generalises** to similar unseen tasks |
| Clean score < 75% | Repair memory does not clearly generalise |
| Merge repair into `index_adapted`, fresh 28-task eval >= 24/28 | **New champion** |

## Path to a clean 28-task promotion

If diagnostic results are promising, the path to a valid new champion is:

1. Copy `memory/raw_v29_repair/repair_records.jsonl` → `memory/raw_adapted/`
2. Rebuild `memory/index_adapted` from the merged raw directory
3. Run the full 28-task benchmark with the updated index
4. If score >= 24/28 with no other changes: new champion

This evaluation is independent because:
- The benchmark tasks were not changed
- The only change is adding repair records to the index
- The repair records were validated on separate test cases, not the benchmark

## Claim boundaries

**Allowed claims after v2.9:**
- "Four failing tasks all receive wrong retrieval from the champion index at k=4."
- "The v2.9 retrieval inspection identifies the specific harmful records by name."
- "A broken task assertion was found in `tree_depth_tuple` (`==3` should be `==4`)."
- "Diagnostic repair achieved `N/28` on the frozen benchmark (DIAGNOSTIC label)."
- "Repair memory generalised to N/5 similar unseen tasks (if clean eval run)."

**Not allowed claims after v2.9:**
- "v2.9 improved the champion score above 23/28."
- "The diagnostic repair result is a clean benchmark evaluation."
- Any promotion claim based solely on diagnostic results.
